## 4. Extracted Content — Ontology & Pipeline Design Principles

> Source: https://community.palantir.com/t/ontology-and-pipeline-design-principles/5481
> This is the single most curriculum-relevant post on the community. Full extraction below.

### Core Concept
The Ontology is the organization's API — a shared layer connecting engineers, business users, and AI agents through business concepts encoding operational processes. Three elements: relevant decision data, action recording, evaluation logic.

### Five-Step Implementation
1. Define user needs — identify decisions, information, nouns (objects), verbs (actions)
2. Verify existing ontology — avoid redundant development
3. Build with placeholder data — mock datasets with PKs and minimal properties
4. Split execution — frontend builds with dummy data; data engineers integrate real data
5. Gather continuous feedback from domain experts

### Object Type Design Rules
- Dedicated owner per object type
- No versioned names (`Message_v2` prohibited)
- No `[tag]` prefixes — use groups
- Intuitive, abbreviation-free naming
- Consistent property naming across platforms
- Minimize properties; avoid redundancy
- Timestamp convention: `{action}_at_timestamp`
- Author convention: `{action}_by_user`
- Maturity status required: Experimental / Active / Deprecated

### Primary Key Rules
- PKs must be strings — no exceptions
- PKs must be inherently unique, derived from object properties only
- All object types require a separate `id` column
- Never infer properties from identifiers
- Composite keys remain unhashed for debugging

### Foreign Key / Link Rules
- FK naming: `{foreign_object_type}_id` or `{link_api_name}_{foreign_object_type}_id`
- Link types use meaningful names (critical for self-referential relationships)

### Action Configuration
- Submission criteria: restrict user groups, validate logic
- Disable "Revert Action" by default (side-effects via Automate/external functions)

### Project Architecture (four categories + sandbox)

| Type | Naming | Purpose | Access |
|------|--------|---------|--------|
| Datasource | `Datasource - {Name}` | Ingest, quality monitoring, raw + clean datasets | Editors + data experts only |
| Data Integration | `Integration - {Name}` | Schema, derived objects, health checks, RVPs | Restricted |
| Ontology | `Ontology - {Name}` | Object types, relationships, views | Editors: engineers/managers; Viewers: end users |
| Application | `Application - {Name}` | Workflows, auxiliary objects, end-user interaction | Collaborative |
| Sandbox | `[sandbox] Name` | Experimentation, training — no business data | Universal creator access |

### RBAC Model
- OWNER: project admin
- EDITOR: build and modify
- VIEWER: consume applications/data
- DISCOVERER: view names and lineage only

## 6. AIP Agent Architecture (extracted from community)

### Components
- **Agent Studio**: ontology/custom context + action/function tools + configurable app state
- **Deployment**: AIP Threads (ad-hoc), AIP Interactive widget (Workshop), Palantir APIs (custom apps)
- **Session Management**: multi-turn via session creation; 24hr expiry (beta)
- **SDKs**: [Python](https://github.com/palantir/foundry-platform-python), [TypeScript](https://github.com/palantir/foundry-platform-typescript)

### Key Community Issues (inform TM-40H/50H content)
- Multi-agent orchestration timeout challenges
- Sub-agent session establishment via OSDK
- Rate limiting with production agent loads
- Agent state persistence and chat log storage
- Streaming API issues in React Native
